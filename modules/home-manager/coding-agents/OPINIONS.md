# Francisco's Opinions

This file is a compact map of Francisco's viewpoints. It is optimized for readability and conciseness.

## Building is easier, so judgment matters more

AI makes building software dramatically easier, which raises the relative importance of knowing what to build.
I want founders to understand real problems, talk to customers, observe decisions, and seek honest feedback before validating their own idea.
I think good ideas start with named people who care about a real problem, not with abstract brainstorming or technology-first excitement.
I favor narrow prototypes, minimal initial scope, and assembling existing building blocks when the goal is to learn quickly.

## Tools should make good choices easy

I value ergonomics because a good architecture that is hard to use correctly still produces performance and maintainability problems.
I like opinionated defaults when they can be centrally optimized, while preserving customization for advanced users.
I prefer terminal-centered workflows with grep, fzf, Neovim-style editing, and low visual clutter.
I value reproducible environments and personal infrastructure because they turn fragile manual memory into repeatable systems.
I prefer clear ownership boundaries between tools over ideological purity about forcing everything through one layer.
I think frameworks and abstractions should earn their complexity by matching the actual problem shape.
I believe terminal and developer tools deserve visual craft, and polish when those details improve comprehension without stealing attention from the user.

## Web development fundamentals

I prefer laveraging the web platform as much as possible, even when using javascript frameworks like React
I like progressive enhancement and server side rendering because they enable resilience and accessible apps 
I prefer using web standard APIs as much as possible over npm libraries

## Code quality and coding style

I prefer simplicity, modular implementations that are read and easy to change, pretty and complex abstracitions
I think code is not cheap. Bad code is the most expensive it has ever been, a codebase that is hard to change locks you out of everything AI can offer
I think codebases naturally drift toward entropy unless senior engineers actively hold the quality bar.
I prefer review cultures that require authors to explain how changes were tested rather than making reviewers personally rediscover every bug.
I believe solo ownership can burn people out and reduce quality when collaboration, shared context, and contributor growth would be better.
